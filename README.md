# Диез Имидж Control Center

Главная ПК-программа / внутренний центр управления экосистемой «Диез Имидж».

## Назначение

Проект должен стать центром управления:

- материалами;
- закупочными ценами;
- товарами;
- услугами;
- заказами;
- клиентами;
- доставкой;
- оплатами;
- статусами;
- будущими интеграциями;
- будущим API/backend для сайта.

## Главное правило

Одна бизнес-система — одна общая база данных.

Центральная база данных находится здесь:

```text
D:\_ProjectHome\diez-data-core
```

Сайт находится здесь:

```text
D:\_ProjectHome\diez-site
```

Этот проект не должен создавать отдельную независимую базу данных и не должен дублировать таблицы заказов, клиентов, товаров, цен, доставок или оплат.

## Архитектурное решение

Desktop:

```text
Tauri 2 + React + TypeScript + Vite
```

API:

```text
Node.js + TypeScript + Fastify
```

База данных:

```text
PostgreSQL из D:\_ProjectHome\diez-data-core
```

Правильный доступ к данным:

```text
Desktop UI -> API -> PostgreSQL
```

DB-секреты, токены доставок, ключи оплат и другие чувствительные данные нельзя хранить в desktop-клиенте.

## Структура проекта

```text
D:\_ProjectHome\diez-control-center
  apps
    desktop
    api
  packages
    shared
  docs
  outputs
```

## Границы текущего каркаса

На этом этапе проект содержит только начальную структуру и документацию.

Сейчас намеренно не создаются:

- рабочее Tauri-приложение;
- рабочий Fastify API;
- package.json и lock-файлы;
- зависимости;
- подключение к PostgreSQL;
- миграции;
- CRUD по заказам, клиентам, оплатам или доставке.

## База и миграции

Все будущие изменения структуры общей базы должны проектироваться и выполняться в `D:\_ProjectHome\diez-data-core`.

В `diez-control-center` нельзя создавать собственные миграции, которые расходятся с общей базой экосистемы.

## MVP-направление

Безопасный первый MVP для ПК-программы — read-only просмотр данных, которые уже есть в общей базе:

- единицы измерения;
- материалы;
- specs материалов;
- закупочные pricing inputs;
- товары;
- услуги;
- specs световых букв.

Заказы, клиенты, оплаты и доставки требуют будущего проектирования схемы в `diez-data-core` и не должны добавляться точечно в этот проект без общей модели.
