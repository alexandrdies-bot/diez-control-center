# Codex Workflow

## Главное правило

Codex не должен придумывать архитектуру, бизнес-логику, модель базы или большие куски кода самостоятельно.

ChatGPT формулирует точное задание, Codex выполняет только указанное.

## Правило документации

В проекте есть 3 главных файла, которые нужно читать в первую очередь:

* `README.md` — главная информация о проекте, назначение, архитектура, навигация, правила.
* `PROJECT_VERSION.md` — текущий статус, версия, что уже сделано, что дальше.
* `CHANGELOG.md` — история изменений по коммитам/этапам.

Папка `docs/` используется только для больших подробных документов, которые не должны раздувать основные файлы.

Новые важные решения сначала кратко фиксировать в 3 главных файлах.

Если создаётся отдельный файл в `docs/`, в `README.md` должен быть понятный список этих документов и зачем они нужны.

## Рекомендуемый интеллект Codex

ChatGPT должен указывать рекомендуемый уровень интеллекта Codex для каждой задачи.

Рекомендуемые уровни:

* Низкий — простая проверка, git status, чтение файла, короткий отчёт.
* Средний — создание документации, небольшие правки, простые конфиги.
* Высокий — код, архитектурные изменения, API, работа с несколькими файлами.
* Очень высокий — сложный аудит, миграции БД, расчёты, интеграции, критичные изменения.

Codex не должен сам выбирать уровень вместо ChatGPT, если уровень уже указан в задании.

## Где писать рекомендуемый интеллект Codex

ChatGPT должен писать рекомендуемый интеллект Codex перед блоком задания, отдельно от копируемого текста.

Пример:

Рекомендуемый интеллект Codex: Средний.

После этого ниже идёт само задание для копирования в Codex.

Служебную строку про интеллект не обязательно копировать в Codex, если пользователь уже вручную выставил нужный уровень в интерфейсе.

## Короткие ответы

В чат можно писать только короткий итог:

* что сделано;
* какие файлы изменены;
* результат проверки;
* git status;
* commit hash, если был commit.

## Длинные отчёты

Если нужен длинный аудит, анализ, список файлов, отчёт или исследование проекта, Codex должен создать временный файл в:

```text
D:\_ProjectHome\diez-control-center\outputs
```

И в чат написать только путь к файлу.

## Длинный код

Если нужен длинный код или большой патч, Codex не должен вставлять его большой простынёй в чат.

Нужно:

1. либо внести точные изменения в указанные файлы;
2. либо создать временный файл с предложением патча в `outputs`;
3. в чат написать только путь к файлу и краткий итог.

## outputs

Папка `outputs` предназначена для временных отчётов и рабочих файлов.

`outputs/` не коммитить, кроме файла:

```text
outputs/.gitkeep
```

## Запрещено

Codex не должен:

* самостоятельно менять архитектуру;
* создавать миграции без отдельного задания;
* менять `diez-site`;
* менять `diez-data-core`;
* писать длинные отчёты в чат;
* писать большие куски кода в чат;
* коммитить временные файлы из `outputs`.
