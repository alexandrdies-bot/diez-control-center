# Site Orders and Chat

## Главный принцип

Заказы с сайта должны попадать в общую систему и отображаться в `diez-control-center`.

Сайт не должен хранить реальные заказы только во frontend/sessionStorage.

Правильная схема:

```text
diez-site -> API -> diez-data-core -> diez-control-center
```

## Заказы с сайта

Когда клиент оформляет заказ на сайте:

* заказ должен создаваться через API;
* заказ должен сохраняться в общей базе;
* заказ должен появляться в ПК-приложении;
* менеджер должен видеть источник заказа: сайт;
* заказ должен иметь статус;
* заказ должен иметь состав заказа;
* заказ должен иметь контакты клиента;
* заказ должен иметь комментарии/файлы/макеты, если они есть;
* менеджер должен иметь возможность обработать заказ из ПК-приложения.

## Будущий чат сайта

На сайте в будущем должен быть чат с клиентом.

Сообщения из чата должны отображаться в ПК-приложении.

Правильная схема:

```text
Клиент на сайте -> сообщение -> API -> база -> ПК-приложение
Сотрудник в ПК-приложении -> ответ -> API -> база -> сайт
```

## Требования к чату

В будущем нужны сущности:

* conversation / диалог;
* message / сообщение;
* customer / клиент;
* assigned_user / ответственный сотрудник;
* channel / канал: site, ozon, manual;
* message status: new, read, answered;
* attachments / вложения;
* timestamps;
* audit/history.

## Роли

Доступ к сообщениям должен зависеть от роли:

* admin видит всё;
* manager видит заказы и чаты;
* designer может видеть сообщения, связанные с макетами;
* production видит только производственные комментарии, если нужно;
* viewer только просмотр.

## Важно

Текущий сайт пока frontend/MVP.

Текущий checkout/sessionStorage — временный.

Перед реальной интеграцией сайта нужны:

* backend API для заказов;
* таблицы заказов в `diez-data-core`;
* таблицы клиентов;
* таблицы сообщений/диалогов;
* авторизация сотрудников;
* права доступа;
* audit log.

## Не делать сейчас

На текущем этапе не создавать эти миграции без отдельного проектирования.

Сейчас нужно только зафиксировать требования.
